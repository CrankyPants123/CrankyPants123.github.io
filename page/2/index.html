Linux——Load Average

发表于 2019-09-09

Load Average

一些探究

简单的说就是用来查看当前系统的负载情况。可以使用uptime或者w/或者top命令来查看。

这个信息依次为：当前时间，当前系统已运行时间，当前已登录用户数，在这个信息的后半部分，有三个数字。意思就是在之前1分钟、5分钟、15分钟之内的系统平均负载。我们可以通过这三个数字来判断系统在指定时间内的负载情况。

所以我们说：平均负载就是单位时间内，CPU的使用率对吗？其实并不全是这样的。我们可以通过man uptime来了解平均负载的详细解释.

简单来说，平均负载是指单位时间内，系统处于可运行状态和不可中断状态的平均进程数，也就是说平均活跃进程数，和CPU的使用率并没有直接的关系。

可运行状态就是我们用ps命令可以看到的处于R（Running或者Runnable状态）的进程。

不可中断状态是正处于内核态关键流程中的进程，并且这些流程是不可中断的。比如常见的等待硬件设备的I/O响应，也就是在ps命令中看到的D状态。比如一个进程向磁盘读写数据的时候，为了保证数据的一致性，在得到磁盘的响应之前，肯定是不能被打断的。加入进程被打断，就容易出现磁盘数据和进程数据不一致的问题。

不可中断状态实际上是系统对进程和硬件设备的一种保护机制。

（思考一下，其实在日常使用电脑的时候，在非高计算的场景下，我们的电脑有时也会出现的比较严重的卡顿。此时引起这种状况的原因往往就是因为磁盘的性能下降，而我们的进程正在向磁盘进行IO操作且处于不可中断状态。一般我们可以通过更换硬盘来解决）

结论

我们可以通过以上的叙述，简单的将平均负载理解为平均活跃进程数，也就是单位时间内的活跃进程数。

这里更加严谨的定义应该是活跃进程数的指数衰减平均值，这里不再做更加深入的探究了。

平均负载的合理区间

假如是双核的CPU，那么当平均负载为2时，表示的意思就是刚好所有的CPU资源都被完全占用，在四核的系统上，就表示CPU有50%的空闲。在单核的系统上则表示，有一半的进程都竞争不到CPU资源。

那我们的平均负载在多少的时候才合理呢？

平均负载与CPU使用率

相信大家和之前的我一样，单纯的把负载就看做为CPU的使用率。但是其实并不是的。追根溯源还是要回到概念：

平均负载是单位时间内处于R状态和不可中断状态的进程数。这其中包含了正在使用CPU的进程，该包括等待CPU和等待IO的进程。

CPU使用率是单位时间内CPU繁忙的情况。和平均负载并不完全一致。

在前文我提到的那个例子就挺合适的，当卡顿的情况发生，此时系统中Runing态的进程基本都处在不可中断的内核态，平均负载会上升，但是CPU其实并不繁忙，反倒是很空闲。因为大家都在等磁盘的响应了。总结一下；

  • CPU密集型的进程，会导致平均负载升高，CPU负载也会升高，此时两者的趋势是一致的，
  • IO密集型的进程，等待IO也会导致平均负载升高，但是CPU的使用率不一定很高，而且有大概率的可能是比较空闲的
  • 大量的CPU进程调度也会导致平均负载升高，此时的CPU使用率也会比较高。

进程线程

僵尸进程

子进程在父进程未调用wait()或者waitpid() 的时候退出。就称为僵尸进程

孤儿进程

父进程退出，但是子进程此时还在运行。那么子进程就会变为孤儿进程，孤儿进程将会由init进程对其进行收集工作。

僵尸进程会导致资源的浪费，但是孤儿进程不会。

  2. 命名管道
  3. 信号
  4. 消息队列
  5. 共享内存
  6. 内存映射
  7. 信号量
  8. 套接字

Unix线程间的通信方式

线程间的通信主要目的是用于线程同步，所以线程没有像进程通信中的用于数据交换的通信机制。

  1. 锁机制：包括了互斥锁、条件变量、读写锁
    互斥锁 提供了以排他方式防止数据结构被并发修改的方法
    读写锁 允许多个线程同时共享数据，而对写操作是互斥的
    条件变量 可以以院子的方式阻塞进程，直到某个特定条件为真为止。对条件的测试是在互斥锁的保护下进行的。条件变量始终与互斥锁一起使用。
  2. 信号量：包括了无名线程信号量和命名线程信号量
  3. 条件变量：类似进程间的信号处理

文件和磁盘

七种文件类型

  1. 普通文件：ASCII还有二进制的都是普通文件。文件第一属性为 [ - ]
  2. 目录文件：属性为 [ d ]
  3. 字符设备或者快设备文件：就是硬盘。属性为 [ b ]
  4. 套接字文件：这类文件通常用来网络数据连接中。属性为 [ s ]
  5. 管道文件：FIFO也是一种特殊的文件类型 。属性为 [ p ]
  6. 链接文件：就是快捷方式 。属性为 [ l ]
  7. 字符设备：串行接口设备。键盘鼠标。属性为 [ c ]

软硬链接

硬链接：多个文件名指向同一索引结点，这种连接就是硬连接。硬连接的作用是允许一个文件拥有多个有效路径名。用户就可以建立硬连接到重要文件，达到防止误删的目的。

软链接：也叫符号链接。类似与windows的快捷方式。实际上是一个特殊的文件。在符号链接中，实际上是一个文本文件，其中包含另一文件的位置信息，

删除原始文件之后，硬链接不受影响，但是软链接会失效。

三大范式

第一范式

1NF

数据表中的所有字段都是不可分割的原子值。字段值还可以继续拆分的，就不满足第一范式。

范式，设计的越详细，对于某些实际的操作可能更好，但是不一定都是好处。

第二范式

2NF

必须是满足第一范式的前提下。第二范式要求，除主键外的每一列都必须完全依赖与主键。如果要是出现不完全依赖，只可能发生在联合主键的情况下。

问题：除过主键以外的其他列，只依赖于主键的部分字段，应该怎么做？

答：拆表。

第三范式

3NF

必须先满足第二范式，除开主键其他列之间不能有传递依赖的关系。

指针

指针和引用的区别

引用本质上就是对象的另外一个名字。int a =10,int &b = a; b = 15相当于a = 15

以下差别都可以被归结为：指针指向一块内存，它的内容是所指内存的地址；但是引用则是某块内存的别名，不能改变指向。

  1. 指针本身就是一个对象，允许对指针的赋值和拷贝。还可以先后指向不同的对象。
  2. 指针无需在定义的时候赋值。引用的时候必须要初始化。
  3. 引用不可以改变指向。
  4. 引用的大小是所指向变量的大小，指针的大小是指针本身的大小
  5. 引用比指针更加的安全。因为不存在空引用。

GET和POST的区别

GET和POST是HTTP请求的两种基本方法。HTTP基于TCP/IP。也就是说GET和POST都是TCP链接。GET和POST所能做的事情也是一模一样的。其实我们在使用GET方法的时候也可以在body中藏点“货”。使用POST的时候也可以在URL中夹带一些参数。所以我们可以得出结论：GET和POST本质上就是TCP链接，并无差别，但是由于HTTP的规定和浏览器/服务器的限制，导致他们在应用的过程中体现出一些不同

  • 最直观的区别是GET把参数包含在URL中，POST通过request body传递参数。
  • GET产生一个数据包，POST产生两个数据包（对于GET方式的请求，浏览器会把http header和data一并发出去。服务器响应200，firefox不同，firefox只发送一个）
  • 对于POST，浏览器会先发送header，服务器响应100 continue，浏览器再发送data，服务器响应200 ok
  • GET的参数在URL中，POST的参数在Request body中。
  • Get只接受ASCII字符，但是POST没有限制。
  • GET比POST更加的不安全，因为GET的参数会直接暴露在URL上。

虽说GET和POST在本质上没有区别，但是浏览器通常都会限制URL长度在2k个字节，大多数服务器最多处理64K大小的URL，超过的部分不会被处理。所以我们要是在URL中藏了“货”，如果数据较多那么就很有可能无法发送，或者被服务器正确接收或处理。

之所以用两次包来传输POST请求，是因为两次包的TCP在验证数据包完整性上是很有优势的。

三次握手过程中可以携带数据吗

结论：前两次握手的时候不可以携带，第三次握手允许携带数据。

RFC193文档中提到了” Data or controls which were queued for transmission may be included”

详细的解释可以参见这篇文章：《TCP 连接建立的三次握手过程可以携带数据吗？》写的非常详细，真的很赞！

具体的意思就是，当TCP_SYN_TECV状态收到合法的二次握手包之后，socket的状态就会立即设置为TCP_ESTABLISHED状态。然后接下来就会处理其中包含的数据。

在大多数情况下，三次握手的过程中是不带数据的。

什么时候带？也可以看上边的文章。嘿嘿，我看不懂。或者说是太笨了。

TCP和UDP

概述：TCP和UDP都属于传输层的协议，其中TCP能提供IP环境下的可靠传输，流量控制和差错恢复的功能。UDP则不为IP提供可靠性和流控、差错控制的功能。一般来说，TCP对应的是可靠性要求高的应用，UDP对应的则是可靠性要求低，传输流量大，传输速度快的的应用。

区别：

  • 面向连接&无连接。TCP有三次握手的连接过程，UDP是和消息的多播发送。
  • 可靠性。TCP利用握手，ACK和重传机制，提供了可靠性保证。而UDP可能丢失。
  • 有序性。TCP利用序列号保证了消息包的顺序交付，到达可能无序，但是TCP会排序。
  • 速度。TCP速度比较慢，因为需要创建连接来保证消息的可靠性和有序性，UDP更加适合做对速度比较敏感的应用，比如多人在线游戏，电视广播，还有在线视频媒体等
  • 重量级。TCP为重量级，UDP是轻量级。体现在元数据的头大小，TCP是20字节，UDP是8字节。

上层协议：

  • 使用TCP协议的上层协议：Telnet、FTP、SMTP、HTTP、HTTPS
  • 使用UDP的上层协议：DNS

使用场景：

  • UDP：需要用到NAT穿透的时候

OSI中常见的协议

  1. 物理层：RJ45,CLOCK,IEEE802.3
  2. 数据链路层：PPP,FR,HDLC,VLAN,MAC
  3. 网络层：IP,ICMP,ARP,RARP,OSPF,IPX,RIP,IGRP
  4. 传输层：TCP,UDP,SPX
  5. 会话层：NFS,SQL,NETBIOS,RPC
  6. 表示层：JPEG,MPEG,ASCII
  7. 应用层：DNS,Telnet,SMTP,HTTP,www,NFS

TTL

数据包在网络传输过程中的最长生命周期，一般以跳为单位。TTL减为0还没有到达的话就会被丢弃。
