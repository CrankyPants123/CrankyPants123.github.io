Linux——kill命令详解

Posted on 2019-09-11

kill命令探究

命令格式

kill [参数] [进程号]

命令功能

发送指定的信号到相应的进程。一共有60多种信号可以供我们来使用。不指定信号的时候默认发送SIGTERM(15)来终止指定的进程。如果仍然无法终止该程序可使用kill -9 pid用来发送SIGKILL信号来强制结束进程。可是使用jobs或者ps命令来查看进程号。root用户可以影响用户的进程，非root用户只能影响自己的进程。

一般来说，在尝试kill掉一个进程的时候是先尝试kill -15 pid如果一段时间后程序仍然在后台运行没有终止的话，再使用kill -9 pid。这种能够防止丢失掉未保存的工作进度。

SIGTERM

此信号是请求一个进程停止运行，是可以被忽略的。进程可以用一段时间来正常关闭，在这段时间内可能会保存一下工作的进度并释放资源。它并不是强制停止的。退出前可以是被阻塞或者回调处理。并且它是Linux缺省的程序中断信号。不过一般来说，当我们去尝试kill掉一个程序的时候，使用SIGTERM不太可能会管用，多半会被阻塞、忽略。

SIGKILL

此信号会强制进程立刻停止运行，而且程序不能忽略此信号，后果是未保存的工作进度都将会丢失。

实例

可以看到我的电脑上正跑着一个校园网的登录脚本。我尝试向对应的进程发送SIGTERM信号，在等待了一段时间之后。程序并没有自动终止。所以可以不难推断出我的信号被程序忽略掉了。

但是当我发送SIGKILL的信号之后。程序就立即被强制终止了。

信号

上述的SIGKILL和SIGTERM是最最常用的两个信号，但是也还有一些其他的常用命令在日常实际使用中其实也会用到。比如：

SIGINT：默认的处理动作是终止进程。

SIGQUIT：默认的处理处理动作是终止进程并且Core Dump.

大型网站架构演化

Posted on 2019-09-11 Edited on 2020-03-09

知识储备：

  • 二八定律：80%的业务访问集中在20%的数据。

大型网站都是从小型网站发展而来，网站架构也是一样从简单到复杂。刚开始的时候访问量少可能只需要一台服务器就绰绰有余了。此时的网站架构基本就是LAMP了。Linux+Apache+MySQL+PHP。此时数据库、程序还有文件都储存在同一台服务器上。

应用服务和数据分离

当访问用户越来越多的时候，开始逐渐出现性能问题。

  • 应用程序需要处理大量业务逻辑，因此需要更加强大的cpu。
  • 数据库需要更快的硬盘和更大的内存。
  • 文件服务器需要更大的容量

单机无法承受这样的性能要求，于是一台服务器被拆分成了三台，应用服务器，文件服务器和数据库服务器。不同特性的服务器承担了不同的服务角色，网站的并发能力大大增强，网站业务进一步发展，并发继续增高。性能问题又再一次面临挑战，此时问题容易出现在数据库服务器上，由于压力太大导致访问延迟，用户体验收到影响。此时就需要对网站的架构再进行优化了。

使用缓存改善网站性能

参见二八定律。既然大部分的业务访问只集中在一小部分的数据上，那么我们就可以考虑把这20%的数据缓存在内存里，缓存高频数据，提高性能。网站的缓存基本可以分为两种：

  • 缓存在应用服务器上的本地缓存
  • 缓存在专门的分布式缓存服务器上的分布式缓存

两者各自的相对优缺点也很明显，本地缓存有限但是很快，分布式的远程缓存没有理论上的容量限制，但是要通过网络就会相对本地缓存慢一些。最高频的小数据放在本地缓存，次之放在分布式的缓存服务器中，接下来的80%就可以放在文件服务器中啦！

使用缓存之后，数据库的访问压力暂时就解决了。不过此时的应用服务器还只是一台。单机能处理的连接有限。访问高峰的时候，应用服务器的性能就又会成为网站的性能瓶颈。

使用应用服务器集群改善网站的并发处理能力

应用服务器实现集群是网站可伸缩架构设计中较为简单成熟的一种。用户的请求通过负载均衡调度服务器来将用户的访问请求分发 到应用服务器集群之中。用户再增加，那就往集群中加入更多的服务器。此时，应用服务器的压力不再成为整个网站的性能瓶颈。那伴随着访问量的继续增长，下一个性能瓶颈是什么呢？

概念

软件测试的目的和原则

目的：验证软件有或者没有问题

原则：遵循软件测试的规范，流程，标准和要求

什么是需求

满足用户期望或者正式规定文档所具有的条件和权能。其中包含了用户需求和软件需求。

用户需求：可以简单理解为甲方的要求，会详细的描述开发人员必须实现的功能。

软件需求：或者叫功能需求，该需求会详细描述开发人员必须实现的软件功能。

软件需求是测试人员进行测试工作的基本依据。

什么是BUG

当程序没有实现其最终用户合理预期的功能要求时，那就是软件错误 or called BUG。

测试用例

测试用例是为了实施测试而向被测试系统提供的一组集合。这组集合中包括：测试环境、操作步骤、测试数据、预期的结果等等。

测试用例可能会遇到的问题

  • 不知道是否较全面的测试了所有的功能
  • 测试的覆盖率无法衡量
  • 对新版本的重复测试比较难以实施
  • 存在大量的冗余测试影响测试效率

软件的生命周期

包括软件产品的设想开始到软件不再使用而结束的时间。可以分为6个阶段，即：需求分析、计划、设计、编码、测试、运维

瀑布模型

start–>需求分析–>计划–>设计–>编码–>测试–>end

瀑布模型在软件工程中占有重要地位，是所有其他模型的基础框架。它的每个阶段之执行一次，因此是线性顺序进行的软件开发模式。

Linux——Load Average

Posted on 2019-09-09 Edited on 2019-09-11

Load Average

一些探究

简单的说就是用来查看当前系统的负载情况。可以使用uptime或者w/或者top命令来查看。

这个信息依次为：当前时间，当前系统已运行时间，当前已登录用户数，在这个信息的后半部分，有三个数字。意思就是在之前1分钟、5分钟、15分钟之内的系统平均负载。我们可以通过这三个数字来判断系统在指定时间内的负载情况。

所以我们说：平均负载就是单位时间内，CPU的使用率对吗？其实并不全是这样的。我们可以通过man uptime来了解平均负载的详细解释.

简单来说，平均负载是指单位时间内，系统处于可运行状态和不可中断状态的平均进程数，也就是说平均活跃进程数，和CPU的使用率并没有直接的关系。

可运行状态就是我们用ps命令可以看到的处于R（Running或者Runnable状态）的进程。

不可中断状态是正处于内核态关键流程中的进程，并且这些流程是不可中断的。比如常见的等待硬件设备的I/O响应，也就是在ps命令中看到的D状态。比如一个进程向磁盘读写数据的时候，为了保证数据的一致性，在得到磁盘的响应之前，肯定是不能被打断的。加入进程被打断，就容易出现磁盘数据和进程数据不一致的问题。

不可中断状态实际上是系统对进程和硬件设备的一种保护机制。

（思考一下，其实在日常使用电脑的时候，在非高计算的场景下，我们的电脑有时也会出现的比较严重的卡顿。此时引起这种状况的原因往往就是因为磁盘的性能下降，而我们的进程正在向磁盘进行IO操作且处于不可中断状态。一般我们可以通过更换硬盘来解决）

结论

我们可以通过以上的叙述，简单的将平均负载理解为平均活跃进程数，也就是单位时间内的活跃进程数。

这里更加严谨的定义应该是活跃进程数的指数衰减平均值，这里不再做更加深入的探究了。

平均负载的合理区间

假如是双核的CPU，那么当平均负载为2时，表示的意思就是刚好所有的CPU资源都被完全占用，在四核的系统上，就表示CPU有50%的空闲。在单核的系统上则表示，有一半的进程都竞争不到CPU资源。

那我们的平均负载在多少的时候才合理呢？

平均负载与CPU使用率

相信大家和之前的我一样，单纯的把负载就看做为CPU的使用率。但是其实并不是的。追根溯源还是要回到概念：

平均负载是单位时间内处于R状态和不可中断状态的进程数。这其中包含了正在使用CPU的进程，该包括等待CPU和等待IO的进程。

CPU使用率是单位时间内CPU繁忙的情况。和平均负载并不完全一致。

在前文我提到的那个例子就挺合适的，当卡顿的情况发生，此时系统中Runing态的进程基本都处在不可中断的内核态，平均负载会上升，但是CPU其实并不繁忙，反倒是很空闲。因为大家都在等磁盘的响应了。总结一下；

  • CPU密集型的进程，会导致平均负载升高，CPU负载也会升高，此时两者的趋势是一致的，
  • IO密集型的进程，等待IO也会导致平均负载升高，但是CPU的使用率不一定很高，而且有大概率的可能是比较空闲的
  • 大量的CPU进程调度也会导致平均负载升高，此时的CPU使用率也会比较高。

进程线程

僵尸进程

子进程在父进程未调用wait()或者waitpid() 的时候退出。就称为僵尸进程

孤儿进程

父进程退出，但是子进程此时还在运行。那么子进程就会变为孤儿进程，孤儿进程将会由init进程对其进行收集工作。

僵尸进程会导致资源的浪费，但是孤儿进程不会。

Unix线程间的通信方式

线程间的通信主要目的是用于线程同步，所以线程没有像进程通信中的用于数据交换的通信机制。

  1. 锁机制：包括了互斥锁、条件变量、读写锁
    互斥锁 提供了以排他方式防止数据结构被并发修改的方法
    读写锁 允许多个线程同时共享数据，而对写操作是互斥的
    条件变量 可以以院子的方式阻塞进程，直到某个特定条件为真为止。对条件的测试是在互斥锁的保护下进行的。条件变量始终与互斥锁一起使用。
  2. 信号量：包括了无名线程信号量和命名线程信号量
  3. 条件变量：类似进程间的信号处理

文件和磁盘

七种文件类型

  1. 普通文件：ASCII还有二进制的都是普通文件。文件第一属性为 [ - ]
  2. 目录文件：属性为 [ d ]
  3. 字符设备或者快设备文件：就是硬盘。属性为 [ b ]
  4. 套接字文件：这类文件通常用来网络数据连接中。属性为 [ s ]
  5. 管道文件：FIFO也是一种特殊的文件类型 。属性为 [ p ]
  6. 链接文件：就是快捷方式 。属性为 [ l ]
  7. 字符设备：串行接口设备。键盘鼠标。属性为 [ c ]

软硬链接

硬链接：多个文件名指向同一索引结点，这种连接就是硬连接。硬连接的作用是允许一个文件拥有多个有效路径名。用户就可以建立硬连接到重要文件，达到防止误删的目的。

软链接：也叫符号链接。类似与windows的快捷方式。实际上是一个特殊的文件。在符号链接中，实际上是一个文本文件，其中包含另一文件的位置信息，

删除原始文件之后，硬链接不受影响，但是软链接会失效。

三大范式

第一范式

1NF

数据表中的所有字段都是不可分割的原子值。字段值还可以继续拆分的，就不满足第一范式。

范式，设计的越详细，对于某些实际的操作可能更好，但是不一定都是好处。

第二范式

2NF

必须是满足第一范式的前提下。第二范式要求，除主键外的每一列都必须完全依赖与主键。如果要是出现不完全依赖，只可能发生在联合主键的情况下。

问题：除过主键以外的其他列，只依赖于主键的部分字段，应该怎么做？

答：拆表。

第三范式

3NF

必须先满足第二范式，除开主键其他列之间不能有传递依赖的关系。

指针

指针和引用的区别

引用本质上就是对象的另外一个名字。int a =10,int &b = a; b = 15相当于a = 15

以下差别都可以被归结为：指针指向一块内存，它的内容是所指内存的地址；但是引用则是某块内存的别名，不能改变指向。

  1. 指针本身就是一个对象，允许对指针的赋值和拷贝。还可以先后指向不同的对象。
  2. 指针无需在定义的时候赋值。引用的时候必须要初始化。
  3. 引用不可以改变指向。
  4. 引用的大小是所指向变量的大小，指针的大小是指针本身的大小
  5. 引用比指针更加的安全。因为不存在空引用。
