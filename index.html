日常充电——博客迁移记

Posted on 2020-02-09 Edited on 2020-02-26

已经有一段时间没有更新博客了，复盘一下我的博客吧。csdn之类的广告我实在是受不了，所以才选择自建。服务之前一直都是使用hexo+NEXT主题，部署在我的阿里学生机上的。访问的时候都是直接使用ip来直接访问。速度很快，这个NEXT主题我个人也一直很喜欢。因为都是一直打算自己一个人使用，也就没有做备案买域名。但是长久使用下来还是发现了很多的缺点。

不足

  • 因为是静态网页，每一次上传的新的博客的时候，都需要重新再生成一次，好的一点是hexo每次做的只是增量更新，耗费的时间并不多。
  • 依赖较高，本地私钥换了一次，就没法再一键部署上去了，还得再重新配置。
  • 需要服务器，略有些成本。
  • 总感觉不放心，毕竟数据无价，保险起见，我还使用坚果云对本地资源做了实时的备份。
  • 本地总是要维护一个hexo的文件夹，虽然文件夹不大，但是我就很烦，因为这样在多个设备上因为没有私钥就没有办法灵活的编辑了。虽然这个问题可以通过配置相同的私钥来解决，但是总归不是一个一劳永逸的解决办法。

改进

  • 项目部署到github pages上，省去服务器的成本
  • github pages国内访问延迟感人，so打算再加上cdn来(全站)加速。
  • cdn没法直接用ip来做加速，需要国内备案的域名。
  • 所以最终的就是博客部署到github pages再用cdn做全站加速，附带还带了域名。
  • 现在的成本就是是cdn和域名的费用了，个人博客没多少流量，域名也很便宜。所以估算下来比我的学生机还是要便宜的。

最终结果

  • 项目部署到了github pages，ping了一下github的服务器在荷兰。so买了一个xyz域名。
  • 备案通过之后申请了免费的ssl证书，正在申请中，cdn加速目前工信部的信息还没有刷新过来

Kafka——-01入门

Posted on 2019-10-11

Kafka

定义

分布式的基于发布/订阅模式的消息队列，主要用于大数据实时处理领域。

MQ（消息队列）的一种传统应用场景就是用来做异步处理，这样做的好处就是能够解耦合。

消息队列的好处

  • 解耦：允许独立的扩展或者修改两边的处理过程，只要确保他们遵守同样的接口约束
  • 可恢复性：一部分组件失效的时候不会影响整个系统，减低了进程间的耦合度，即使有处理消息的进程挂掉，加入队列的消息仍然可以在系统恢复后被处理。
  • 缓冲（削峰）：有助于控制和优化数据经过系统的速度，解决生产消息和消费信息的处理速度不一致的情况。
  • 灵活性&峰值处理能力：在访问量剧增的情况下，应用仍然需要继续发挥作用。但是这样的突发流量并不常见。如果以能处理这类峰值访问为标准来投入资源随时待命是一种巨大的浪费，使用消息队列就能使关键组件顶住突发的访问压力，不会因为突然的超负荷的请求而完全崩溃。
  • 异步通信：很多时候也不需要立即处理消息。消息队列提供了异步处理机制，允许用户把一个消息放入队列，但并不立即处理它。想向队列中放入多少消息就放多少，然后在需要的时候去处理它们。

消息队列的两种模式

点对点模式

一对一，消费者主动的去拉取数据，消息收到后消息清除。

发布/订阅模式

一对多，消费数据后不会清除消息。消费者的消费速度可以由消费者来决定。有消费者主动拉取数据和消息队列主动推送数据两种方法的区别。消费者去主动拉取的话有个确定就是需要维护一个常轮循，比较消耗性能。由消息队列来推送的话又有可能出现消费者处理不完，浪费数据的情况。

nosql介绍

NoSQL：一类新出现的数据库（not only sql)特点：

  • 不支持SQL语法。
  • 储存结构跟传统关系型数据库中的关系表完全不同，nosql中储存的数据都是KV形式。
  • NoSQL没有通用的语言，不像SQL一样基本统一。每种nosql数据库都有自己的api和语法，以及擅长的业务场景
    • Mongodb
    • Redis
    • Hbase hadoop
    • Cassandra hadoop

NoSQL和SQL数据库比较

  • 适用场景不同：sql数据库适合用于关系复杂的数据查询场景，nosql反之
  • 事务的支持：sql对事务的支持很完善，nosql基本不支持事务。

Redis简介

Redis是一个使用C来编写的开源程序，支持网络，可基于内存亦可持久化的日志型，Key-Value数据库。并且支持多种语言的API。

Redis是NoSQL阵营中的一员，通过多种键值数据类型来适应不同场景下的储存需求，借助一些高层级的接口使用其可以胜任缓存。队列系统的不同角色。

特性

  • 支持数据的持久化，可以将数据存在磁盘中，重启的时候再次加载进行使用。

  • 不仅支持kv数据，还提供了list、set、zset、hash等数据结构的存储。

  • 支持数据的备份，即master-slave模式的数据备份。

##优势

  • 性能高：毕竟这东西的介质是RAM，读110000次/s，写81000次/s
  • 数据类型丰富：看上边
  • 原子：Redis的所有操作都是原子的，同时还支持对几个操作全并后的原子性执行
  • 特性丰富：Redis还支持publish/subscribe、通知、key过期等等特性

应用场景

  • 用来做缓存。

  • 在某个场景下替代传统数据库。

  • 在大型系统中，实现特定的功能。比如session共享，购物车。

Linux——用户态和内核态

Posted on 2019-09-13 Edited on 2019-09-16

用户态和内核态

在信号的那篇blog中曾经写到，当接收到硬件中断的时候，CPU会从用户态切换到内核态来处理硬件中断。那么问题来了——为什么CPU非得要切换到内核态才能来处理硬件中断呢？

概念

首先来说说概念：

内核态：CPU可以访问内存的所有数据，包括了外围设备，CPU也可以将自己从一个程序切换到另一个程序。此时所占用的处理机是不允许被抢占的。

用户态：只能访问受限的内存，而且不允许去访问外围设备，占用CPU的能力也被剥夺。CPU资源可以被其他程序获取或者抢占。

看完概念总结一下：可以认为内核态就是一种CPU的特权状态，只有在这个状态下，CPU才可以执行这个特权态下允许执行的指令，访问这个特权态才允许访问的资源，这个和当前的进程无关。

原因

由于需要限制不同CPU之间的访问能力，防止他们获取别的程序的内存数据，或者外围设备的数据，并发送到网络，所以，CPU划分出了两个权限等级——用户态和内核态。

切换

所有的用户程序都是运行在用户态的，但是有的时候程序确实需要做一些内核态事情，比如从硬盘读取数据，从键盘获取输入等，所以此时程序就需要请求操作系统以操作系统的名义来执行这些操作。

​ 此时，我们就需要这样一个机制：用户态程序切换到内核态，但是不能控制在内核态中执行的指令。

这种机制叫做系统调用，在CPU中的是实现称为陷阱指令

系统调用

系统调用运行在内核态，而且系统调用的库函数和普通函数调用非常的相似，只是系统调用由操作系统的内核来提供，运行与内核核心态，而普通的库函数调用由函数库或者用户自己提供，运行于用户态。系统调用本质上来说还是通过系统调用来实现的，而异常和中断的处理机制基本一致。

特权级别

其实我们不难推测出，创建一个新的进程对于任何一个操作系统来说，都是非常核心的功能。其中涉及到物理资源的分配，拷贝设置页目录表等等，显然不能随便就交给一个程序就可以去做的。于是，特权级别的概念自然而然就被引出了。这样才能做到集中管理，减少有限资源的访问和使用冲突。

为了支持特权级别，举例牙膏厂 X86架构的CPU就在硬件上做了很多的支持。其中CPU共有0~3总共四个特权级别。0最高3最低。对于Unix like系统来说，只利用到了0和3级别。

所以自然而然的就可以解释，用户态的程序，CPU就是才特权级别3状态下运行的。内核态就是0状态。

我们知道fork()实际上是以系统调用的方式来完成相应功能的。具体的工作由sys_fork来负责实施。但是testfork()是不能直接调用sys_fork(),前者是工作在用户态，属于用户态程序。后者是工作在内核态，属于内核态程序。

当我们在系统中执行一个程序，大部分时间是运行在用户态下的，在其需要操作系统帮助完成其他它没有权力和能力完成的工作时就会切换到内核态，比如testfork()最初运行在用户态进程，当它调用fork()最终触发sys_fork()的执行时，就切换到了内核态。

切换的方式

  1. 系统调用：这是用户态程序主动要求切换到内核态的一种方式。用户态进程通过系统调用申请使用操作系统提供的服务程序完成工作。比如上述的例子中fork()实际上就是执行了一个创建新进程的系统调用。而操作系统调用的机制核心还是使用了操作系统为用户特别开放的一个中断来实现，比如Linux的int 80h中断。
  2. 异常：CPU在执行运行在用户态程序的时候，遇到了一些不可知的异常，这就会触发一个异常的中断，跳转到了内核态。比如缺页异常。
  3. 外围设备的中断：当外围设备完成用户请求的操作后，会向CPU发出相应的中断信号。如果之前执行的指令是用户态下的程序，那么这个转换的过程自然也就发生了用户态到内核态的切换。比如硬盘读写操作完成。系统会切换到硬盘读写的中断处理程序中执行后续操作等。

切换操作

  1. 从当前进程的描述符中提取其内核栈的ss()的esp0的信息。
  2. 使用ss()和esp0指向的内核栈将当前进程的cs,eip,eflags,ss,esp信息保存起来，这个过程也完成了由用户栈道内核栈的切换过程，同时保存了被暂停执行的程序的下一条指令。
  3. 将之前向中断向量检索得到的中断处理程序的cs,eip信息装入相应的寄存器。开始执行中断处理程序，这个时候就转到了内核态的程序执行了。

特权指令

指的就是在内核态时运行的指令，此时：

  • 对内存空间的访问范围基本不受限制，不仅能访问用户的储存空间，也能访问系统存储空间。
  • 特权指令只允许操作系统使用，不允许其他应用程序使用，否则就很有可能导致系统异常崩溃等。

特权指令有：

  • 启动I/O
  • 内存清零
  • 修改程序状态字
  • 设置时钟
  • 允许/禁止中断
  • 停机

非特权指令

指的就是在用户态时运行的指令，一般应用程序所使用的都是非特权指令，它只能完成一般性的操作和任务，不能对系统中的硬件和软件直接进行访问，对内存的访问也只仅限于用户空间。

非特权指令有：

  • 控制转移
  • 算数运算
  • 访管指令
  • 取数指令

Linux——信号

Posted on 2019-09-11 Edited on 2019-09-19

信号

今儿面试的时候被面试官的kill给噎住了，感觉很难受。于是想写下kill，然后就写到了信号。于是有了这篇blog。

基本概念

例举一个典型的场景：

  1. 用户输入命令，在shell启动了一个前台进程。
  2. 用户按下了Ctrl + C组合键。此时在这个键盘输入产生了一个硬件中断。
  3. 如果CPU正在执行当前这个进程的代码，则该进程的用户空间代码暂停执行。CPU从用户态切换到内核态道处理硬件中断。
  4. 终端的驱动程序将Ctrl+C解释成一个SIGINT信号，记在该进程的PCB中（也可以说成发送了一个SIGINT信号到该进程).
  5. 在某个时刻，内核要返回到用户空间继续执行代码了，首先他会处理PCB中记录的信号，发现有一个SIGINT的信号在等待处理，而且这个信号的默认处理动作是终止进程，所以会直接终止进程，而不再返回它的用户空间代码执行。

信号想对于进程的控制流程来说是异步的。也就是说前台进程在运行过程中随时都有可能因为收到SIGINT信号而终止。也可以说信号就是软件层面上实现的软中断。

只有前台的程序才能接受到Ctrl+C这样的控制键产生的信号。

进程收到的所有信号，都是由内核负责发送和处理的。

使用kill -l命令可以查看系统定义的信号列表

信号的产生

通过终端按键产生信号

比如当我们按下Ctrl+C就会给前台程序发送SIGINT信号，SIGKILL的默认处理动作是终止进程，SIGQUIT的默认动作是终止进程并且Core Dump。

常见控制键产生的信号：

Ctrl+c：SIGINT信号。代表终止进程，默认的动作是终止进程。

Ctrl+\：SIGQUIT信号。代表退出进程，默认动作是终止进程并且Core Dump。

Ctrl+z：SIGSTP信号。代表终止进程，默认动作是终止进程，和SIGINT不同的是SIGSTP不能被捕获。

Ctrl+d：不发送信号，但是能表示一个特殊的二进制值——EOF。

Core Dump

当一个进程要终止的时候，可以选择把进程的用户空间内存数据全部保存到硬盘上，文件名通常是core，这个过程叫做Core Dump。事后我们可以使用core进行Debug等操作。

通过系统调用函数

就是用kill命令。kill命令是通过调用kill函数来实现的，kill函数可以给指定的进程发送指定的信号。除过kill命令之外，还有一个raise函数，可以给当前进程发送指定的信号。还有个abort函数可以让程序直接异常终止。

软件条件产生

SIGPIPE是一种由软件条件产生的信号，这里会主要讲一下alarm函数和SIGALRM。

alarm函数

注意：

  • 函数的返回值是0或者之前设定还剩余的秒数。
  • 一个进程只能有一个闹钟时间，如果在调用alarm之前已经设定过闹钟，那么之前的设定会被新设定的值取代。
  • 信号由内核产生，由于进程调度的延迟，所以进程得到信号到处理还是需要一些时间的。
  • 如果second的值是0，代表的是取消之前设定的值。
  • 出错返回 -1

阻塞信号

相关概念

  • 实际执行信号的处理动作称为信号递达（Delivery）
  • 信号从产生到递达之间的状态称为信号未决（Pending）
  • 进程可以选择阻塞（Block）这个信号。
  • 被阻塞的信号产生时保持在未决状态，直到进程解除对此信号的阻塞，才执行递达的动作。
  • 阻塞和忽略并不同，信号只要被阻塞就不会递达，但是忽略是递达之后可选的一种处理动作。
